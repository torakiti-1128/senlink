# 要件定義書：SenLink（センリンク）

本書はプロジェクト憲章（`docs\01_PLAN\PROJECT_CHARTER.md`）を前提とし、SenLink が満たすべき機能要件・非機能要件を定義する。


## 1. 前提・範囲

- 対象組織：筆者が卒業した専門学校
- 対象ユーザ：
  - 学生
  - 教員
  - 管理者（管理者権限を持った教員）
- 本バージョンでは「就職活動に関する情報一元管理」と「進捗・タスク管理」を主対象とし、
  他システム（学生情報DB、成績管理等）との連携は範囲外とする。

※本資料における「求人」とは、企業説明会／インターンシップ／採用試験を含む総称とする。


## 2. 目的（要約）

- 学生の求人機会の見逃しを低減し、適切な求人選択を支援する。
- 教員の事務負担（進捗確認／集計／催促）を削減し、キャリア相談に集中できるようにする。
- 指導履歴を蓄積し、教員間でナレッジ共有できる状態をつくる。
- 学生が就職活動タスクを把握・管理できるよう支援する。


## 3. 機能要件

### 3-1. 学生向け機能

#### F-1：求人一覧・検索・詳細
- 学生は、システム内に登録された「これから開催される求人（企業説明会／インターンシップ／採用試験）」を一覧で閲覧できること。
- 一覧には、少なくとも以下の情報が含まれること：
  - 求人種別（企業説明会／インターンシップ／採用試験）
  - 案件名
  - 企業名
  - 開催日
  - 応募締切日
  - 勤務地・職種・特徴などのタグ
  - 学生にとっての状態（例：応募可／応募済み／満席／締切 など）
  - 教員からのレコメンドがある場合は、そのことが分かる表示
- 学生は、以下の条件で求人を検索・絞り込み・並び替えできること：
  - フリーワード（企業名・案件名・キーワードなど）
  - 求人種別（すべて／企業説明会／インターンシップ／採用試験）
  - 開催日（期間指定）
  - 勤務地
  - 職種
  - 特徴（例：短期／長期／オンライン対応 など）
  - 並び順（開催日が近い順／応募締切が近い順／新着順 など）
- 検索結果件数が分かること（例：「○件ヒット」）。
- 学生は、一覧から求人を選択し、その詳細情報を閲覧できること。
- 詳細には、少なくとも以下の情報が含まれること：
  - 求人種別
  - 案件名
  - 企業名
  - 開催日時（開始〜終了）
  - 開催形式（対面／オンライン／ハイブリッド）
  - 開催場所（オンラインの場合は参加用リンク等）
  - 定員と現在の応募状況（残席数が分かること）
  - 募集要項の詳細本文
  - 応募締切とキャンセル期限
  - 緊急連絡先（当日の連絡先）
  - 企業の基本情報（所在地・Webサイト等）
  - タグ（勤務地・職種・特徴など）
- 開催日が過ぎた求人を、学生が詳細情報まで参照できること（過去の活動履歴として閲覧可能）。
- 学生は、詳細画面から求人に応募できること。
  - 応募済みの場合は、再応募できないこと。
  - 応募できない条件（例：満席・締切後・過去開催など）の場合は、応募操作ができないことが明確に分かること。
- 学生は、求人を「ブックマーク（気になる）」として登録／解除できること。
  - ブックマーク状態は一覧および詳細の両方で確認できること。
- 以下のいずれかに該当する求人は「応募不可」として扱われること：
  - 定員に達している（満席）
  - 応募締切日を過ぎている
  - 学生がすでに応募済みである
- 応募不可の求人であっても、詳細情報の閲覧は可能であること。

#### F-2：求人への応募とToDo管理
- 学生は、自身の就職活動状況のサマリを確認できること。
  - 応募数（説明会／インターンシップ／採用試験／内定など）の集計
  - 直近のイベント（次に参加予定の求人）
  - 要対応のタスク数（期限切れ／今日期限など）
  - 全体のタスク進捗率（完了タスク数／全タスク数）
- 学生は、自身が応募した求人の進捗状況を一覧で確認できること。
  - 求人種別（説明会／インターンシップ／採用試験）
  - 開催日時（未来／過去の区別を含む）
  - ステータス（参加前／参加済／辞退申請中／辞退済 等）
  - 進捗指標（例：ToDo 完了率、残タスク数 等）
- 学生は、求人ごとに「現在対応すべきタスク」を確認できること。
  - タスク名／内容／期限日などの情報を含むこと。
  - 期限が近い・過ぎているタスクは、視覚的に強調されること。
- 学生は、タスクを完了として更新できること。
- 学生は、応募した求人を条件で絞り込み／並び替えできること。
  - 状態（参加前／参加済み／すべて）
  - 求人種別
  - 並び順（進捗が遅れている順／期限が近い順／開催日順 など）
- 学生は、応募済みの求人について辞退（キャンセル）の申請ができること。
  - 辞退は即時確定ではなく、教員による承認が必要な申請として扱われること。
  - 辞退申請中の求人は、一覧上でその状態が分かるように表示されること。
- 重要な状態（期限切れタスク、辞退申請中など）は強調表示されること。
- 多数の応募が存在しても、主要な情報が一目で把握できるレイアウトであること。

#### F-3：学校への申請管理（申請ダッシュボード）
- 学生は、学校側に提出可能な申請の種類を一覧から選択できること。
  - 面談予約／書類発行／内定報告
  - 申請種別は教育機関の運用に応じて変更可能であること
- 学生は、選択した申請種別ごとに必要な情報を入力して送信できること。
  - 種別に応じて入力項目が異なること
  - 入力項目のバリデーションルールは後続工程（設計）で定義する
- 学生は、過去に提出した申請の履歴を確認できること。
  - 少なくとも以下の情報を含むこと：
    - 申請日時
    - 申請種別
    - 概要（申請内容の要約）
    - ステータス（申請中／承認／却下／下書き等）
- 学生は、各申請の承認状況を確認できること。
  - 教員や管理者からのコメントやフィードバックが含まれる場合はそれを閲覧できること
- 申請ステータスは、教育機関側の処理により更新されること。
  - 更新結果（例：承認／却下）は学生側に通知されること（通知手段は後続工程で定義）
- 申請履歴は時系列で管理されること
- 承認／却下は学生が操作できないこと（学校側の権限）
- 申請種別・承認フローは教育機関の運用に従うこと

---

#### F-4：学生マイページ
- 学生は、自身に関する基本情報を参照できること。
- 基本情報には、少なくとも以下の情報が含まれること：
  - 氏名
  - 学籍番号
  - 所属（学科・クラス）
  - 生年月日
  - 性別
  - 学校メールアドレス
- 上記のうち、学校側で管理される項目については学生が編集できないこと。
- 学生は、就職活動に必要な自己情報を編集できること。
- 編集可能な項目には、少なくとも以下の情報が含まれること：
  - 希望職種
  - 希望勤務地
  - 自己PR
  - 資格・スキル
  - ポートフォリオURL
- 希望職種および希望勤務地は、求人推薦や検索最適化に利用されることを前提とする。
- 学生は、自身の就職活動状況を設定できること（例：就職活動中／活動終了）。
- 就活ステータスは、教員側の管理画面に反映され、指導対象判定に利用されること。
- 学生は、システム利用に関する設定を管理できること。
- 設定には、少なくとも以下を含むこと：
  - パスワード変更
  - 通知設定（メール通知／プッシュ通知等）
  - ログアウト
- プロフィール更新後は、変更内容が即時反映されること。
- アカウントに紐づく識別情報（メールアドレスなど）は、システムの認証基盤に依存するため変更権限は別途扱うこととする。

### 3-2. 教員向け機能

#### F-5：教員用ダッシュボード
- 教員は、自分が担当するクラスを選択し、そのクラスの就職活動状況サマリを確認できること。
  - 就活中の学生数
  - 内定獲得者数および内定率
  - 一定期間内に活動している学生の割合（活動率）
  - 未内定者数
- 教員は、クラス内の「学校側対応が必要な申請」の件数を確認できること。
  - 書類発行依頼
  - 面談／面接練習の予約
  - 内定報告
  - 上記の合計件数
  - 対象をクリックすることで、申請管理画面へ遷移し、該当クラス・種別・ステータスで絞り込まれた一覧を確認できること。
- 教員は、支援が必要と思われる学生を一覧で確認できること。
  - 例：
    - 一定期間ログインしていない学生
    - 一定期間、応募やタスク完了などの活動がない学生
    - 期限を過ぎたタスクを抱えている学生
  - 各学生について、少なくとも氏名・学籍番号・状況（最終ログイン日／未完了ToDo件数 等）が分かること。
  - 学生を選択することで、その学生の詳細画面に遷移できること。
- 教員は、クラス内の最近の活動ログを時系列で確認できること。
  - 例：
    - 内定が承認された
    - 求人に応募した
    - 面談を申請した
  - ログから、学生詳細画面や求人詳細画面へ遷移できること。
- 教員は、今後の面談・指導予定および面談申請状況を確認できること。
  - 直近一定期間内の確定済み面談予定（日時・学生・担当教員・場所など）
  - 承認待ちの面談申請件数
  - 必要に応じて申請管理画面へ遷移できること。
- 教員は、自分自身の対応状況に関する指標を確認できること。
  - 自分が対応すべき未処理件数（書類／面談予約／内定報告など）
  - 直近一定期間の平均対応時間（申請〜対応完了まで）
  - 対応が遅延している申請の一覧（例：申請から一定時間以上経過しているもの）
 - 教員は、自身が担当するクラスおよび学生の情報のみ閲覧できること（権限に応じた制限）。
 - ダッシュボードに表示される数値・一覧は、最新のデータに基づいていること。
 - 多数の学生・申請が存在する場合でも、優先度の高い情報（支援が必要な学生・未処理申請など）が視認しやすい構成であること。

## 4. 非機能要件（抜粋）

### 4.1 性能・スケーラビリティ
- 主要な読み取り系APIのレスポンスタイムは、通常時 P95 で 200ms 以内を目安とする。
- 将来的な学生数・求人数の増加に耐えられるよう、DB 設計は N+1 クエリおよびフルテーブルスキャンを避けること（`EXPLAIN` により検証）。

### 4.2 品質
- テストカバレッジ 80%以上（行カバレッジ）。  
- 関数単位の循環的複雑度 10 以下を目安とする。  
- 静的解析（Lint / Formatter / 型チェック）エラーゼロを CI で担保する。

### 4.3 運用・保守
- Docker により、ローカル・検証環境・本番環境の構成差異を最小化する。
- ソースコード管理は GitHub を使用し、すべての変更は Pull Request 経由とする。
- Pull Request は必ず GitHub Issues と紐づける。

### 4.4 セキュリティ
- 学内ネットワーク以外からアクセスできない。
- 学生・教員の区別を行い、閲覧可能な情報を適切に制限する（権限モデルの詳細は設計フェーズで定義）。
- パスワード等の機密情報は平文で保存しない。
- アカウント登録ではワンタイムパスワード認証方式を用いる。
- セッション管理ではタイムアウト機能を持たせる。



## 5. 対象外（非スコープ）

- 企業側のログイン画面および求人登録機能（現時点では学校側が求人情報を登録する前提）。
- 他学内システム（成績管理・学生管理DB等）との連携。
- 既存チャットツール（Teams 等）との双方向連携。
