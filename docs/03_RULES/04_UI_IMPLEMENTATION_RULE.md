# UI実装規約 (UI Implementation Rules)

このドキュメントは、SenLinkプロジェクトにおけるフロントエンド（UI/UX）の実装指針を定めたものである。高品質で一貫性があり、保守性の高いユーザーインターフェースを構築するためのルールを規定する。

---

## 1. 技術スタック

| 項目 | 採用技術 | 備考 |
| :--- | :--- | :--- |
| **Framework** | **Next.js (App Router)** | TypeScriptを必須とする。 |
| **Styling** | **Vanilla CSS / CSS Modules** | **Tailwind CSSは原則禁止**。独自の意匠性を優先する。 |
| **Components** | **shadcn/ui** | 基本コンポーネントの土台として利用するが、スタイルはカスタマイズする。 |
| **Icons** | **Lucide React** | 一貫したアイコンセットを使用する。 |
| **Form** | **React Hook Form + Zod** | 型安全なバリデーションを徹底する。 |
| **State** | **React Context / Zustand** | 必要最小限のグローバル状態管理に留める。 |

---

## 2. ディレクトリ構成

`src/SenLink.App/src/` 配下を以下の通り組織化する。

*   `app/`: ルーティングおよびページコンポーネント。
*   `components/`:
    *   `ui/`: shadcn/uiベースの基本パーツ（ボタン、入力等）。
    *   `brand/`: ロゴやブランド固有の意匠コンポーネント。
    *   `features/`: 機能特化型のコンポーネント（例: `auth/`, `school/`）。
*   `hooks/`: 共通カスタムフック。
*   `lib/`: 外部APIクライアント、ユーティリティ。
*   `locales/`: 多言語・メッセージ定義ファイル（JSON）。
*   `types/`: 共通型定義。

---

## 3. コーディング規約

### 3.1. コンポーネント設計
*   **関心の分離**: ビジネスロジック（API呼び出し等）はページまたはカスタムフックに寄せ、コンポーネントは可能な限り純粋なプレゼンテーションに徹する。
*   **リアクティブ性**: `useWatch` を活用し、フォームの状態変化に応じた動的な表示制御を徹底する（非リアクティブな `getValues` への過度な依存を避ける）。
*   **型定義**: Propsには必ずインターフェースを定義し、`React.FC` ではなく関数宣言（`function Component(...)`）を推奨する。

### 3.2. スタイリング
*   **Vanilla CSSの優先**: セマンティックなクラス名（例: `.onboarding-card`）を付与し、グローバル変数（CSS Variables）を用いて色、フォント、余白を一括管理する。
*   **モダンな意匠**: グラデーション、柔らかな影（Soft Shadows）、一貫した角丸（Rounded Corners）を用い、現代的で「生きている」感覚を与える。
*   **レスポンシブ**: モバイルファーストで設計し、`@media` クエリを用いてデスクトップまで最適化する。

---

## 4. エラーハンドリングとメッセージ管理

### 4.1. Operation名ベースのメッセージ
*   APIから返却される `Operation` プロパティをキーとして、`locales/ja/errors.json` からメッセージを解決する。
*   ハードコーディングされた日本語文字列をコンポーネント内に記述することを禁止する。

### 4.2. フィードバック
*   **Loading**: 非同期処理中は必ず `Loader2Icon` 等を表示し、ボタンを `disabled` にする。
*   **Toast**: 処理の成功・失敗は `sonner` を用いたトースト通知で即座にフィードバックする。

---

## 5. 品質保証

*   **アクセシビリティ**: 適切なHTMLタグ（`main`, `section`, `nav` 等）と `aria-label` を使用する。
*   **ビルド確認**: コミット前に必ず `npm run build` が通ることを自ら確認する。
*   **一貫性**: 既存の `onboarding/page.tsx` 等のパターンを模範とし、大幅な意匠の逸脱を避ける。
