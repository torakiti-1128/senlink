# プロジェクト憲章：SenLink (センリンク)

## 1. プロジェクト概要
専門学校における「学生と教員」「学生と企業」の繋がり（Link）を最適化し、就職活動における情報一元化とプロセス管理を効率化する就職活動支援システム。


## 2. 背景と目的

### 2.1 開発背景：学生時代の開発のリベンジ
本プロジェクトは、学生時代に開発した就職活動支援システムの「再構築」である。過去のプロジェクトでは以下の4つの問題により、継続的な運用／保守が困難な状態に陥った。

- **上流工程の欠如**：要件定義／設計をスキップした「コードファースト」な開発
- **設計の不在**：ビジネスロジックがViewに混在し、ソースコードが重複する低い保守性
- **品質管理の欠如**：テストコードが皆無であり、修正に伴う退行（デグレ）が頻発
- **開発プロセスの欠如**：バージョン管理ルールやタスク管理がなく、変更履歴が追跡不能

### 2.2 プロジェクトの目的
実務（運用保守／テスト工程）で培った「品質重視の姿勢」と「エンジニアリング能力」を証明する。

- **技術力証明**：
    - モジュラーモノリス構成による責務の明確化
    - テスト駆動開発（TDD）の実践による、仕様が担保された堅牢なコードベースの構築
- **管理能力証明**：
    - チケット駆動開発、Gitフロー、ドキュメント管理の徹底
- **適応力証明**：
    - **仕様ファーストの徹底**：AIを開発パートナーとし、仕様書からコードを生成／同期させる開発スタイル（仕様駆動開発）の実践


## 3. SenLinkについて
私が卒業した専門学校における就職活動では、情報が一元管理できておらず、学生の活動漏れや教員の過度な事務負担を引き起こしていた。  この課題を解決するため、SenLinkは以下の4点に取り組む。

※本資料に記載している「求人」は、企業説明会／インターンシップ／採用試験を含む総称である。

### ① 学生が求人の応募機会を逃さないようにする
* **課題**：求人がTeams、メール、紙媒体に分散し、把握が困難だった。 学生が自分に合う求人を見逃すリスクが高い。
* **解決**：全チャネルの求人を一元管理し、検索性を高めマッチング精度を向上させる。

### ② 教員がキャリア相談に集中できるようにする
* **課題**：学生の進捗を個別にヒアリングし、Excelで手動集計／催促していた。事務作業に追われ、本来注力すべき「キャリア相談」のリソースが不足。
* **解決**：学生の行動ログから進捗を自動集計・可視化し、定型業務をシステムで代替する。

### ③ 学生が就活タスクを管理できるようにする
* **課題**：応募した求人を、経験の浅い学生自身が手動で管理していた。期限失念や準備不足が発生。
* **解決**：ToDoを自動生成し、システムが学生を支援する。

### ④ 指導ノウハウを組織に蓄積できるようにする
* **課題**：添削依頼や面接フィードバックが、チャットや口頭のやり取りで完結している。指導履歴が蓄積されず、指導の質が担当教員に依存する。
* **解決**：データベースに履歴を保存し、組織内でのナレッジ共有を可能にする。


## 4. 開発スコープ

### 4.1 学生向け機能
- **求人を探す**：検索、応募、ブックマーク機能
- **ToDo生成**：応募後に必要な作業をToDoリストとして学生に提供
- **申請管理**：書類添削・面接予約・合否報告のワークフロー
- **通知システム**：期限間近のタスクや新規求人の通知

### 4.2 教員／管理者向け機能
- **ダッシュボード**：担当クラス全体の活動状況、内定率のリアルタイム可視化
- **学生詳細管理**：個別の活動履歴、添削依頼等のフィードバック
- **マスタ管理**：求人情報の登録、教員／クラスの管理、設定値変更
- **システム管理**：システムの稼働状況を可視化


## 5. 成功指標

### 5.1 技術品質（必須）
- **テストカバレッジが80%以上**：`pytest` 等で計測し、数値で証明する。
- **循環的複雑度が10以下**：`radon` 等で計測し、数値で証明する。
- **静的解析エラーゼロ**：Linter/Formatter がパスしない限りマージしない。

### 5.2 管理品質（必須）
- **チケットとコードの紐付け**：すべてのプルリクエストが GitHub Issues に紐付いている。
- **振り返りの実施**：開発後に「良かった点」「改善点」「技術的負債」を記録する。

### 5.3 プロダクト品質（プラスアルファ）
- **Lighthouse Performance スコア90以上**：主要ページにおける表示速度を数値で担保する。
- **スケーラビリティを考慮したDB設計**：クエリにおいて、N+1クエリおよびフルテーブルスキャンが発生しないことを検証済みとする。客観的根拠を提示できる場合は例外とする。


## 6. 技術スタック方針
技術選定は「保守容易性」「変更容易性」「開発速度」を優先した。

- **Frontend：Next.js + shadcn/ui**  
  初期表示性能と開発効率を両立するため。

- **Backend：FastAPI + RabbitMQ**  
  型安全な高速API実装と、非同期処理の疎結合化のため。

- **Architecture：モジュラーモノリス**  
  個人開発規模で責務分離と保守性を両立するため。

- **Reverse Proxy：Apache**  
  HTTPS 終端とリバースプロキシとして既存 VPS 環境と最も親和性が高いため。

- **Infrastructure：Docker + VPS（Xserver） + Supabase（Postgres / Storage）**  
  再現性／低コスト運用と、DB/ストレージのみのマネージド活用のため。

- **AI Assistant：Claude Code**  
  仕様駆動開発に適し、ターミナル統合で効率的に開発できるため。
